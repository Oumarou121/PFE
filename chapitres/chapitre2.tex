\chapter{Analyse et spécification des besoins}

\section{Introduction}

Avant de taper la moindre ligne de code, il a fallu traduire les besoins métier en spécifications techniques. Dans un projet comme SIRH-Doc, l'analyse est l'étape où l'on décide comment l'application va \og penser \fg{}. 

L'un des plus gros enjeux a été de définir une structure assez souple pour s'adapter à n'importe quelle table de données RH, tout en restant assez simple pour un utilisateur qui n'est pas informaticien. Ce chapitre détaille les acteurs du système, les fonctionnalités indispensables et les choix de modélisation qui ont guidé mon développement.

\section{Qui utilise le système ? (Les Acteurs)}

J'ai identifié trois profils types, chacun ayant un périmètre d'action bien précis. Il est important de noter que ces acteurs sont les \textbf{utilisateurs administratifs} (ex: Secrétaire Général (SG), Chef de Département (CDD), Responsable Master) qui pilotent la plateforme pour gérer diverses entités :

\begin{itemize}
    \item \textbf{Le Super Administrateur} : C'est le \og maître d'œuvre \fg{}. Il configure les établissements (tenants), crée les modules métier et définit quelles tables de la base de données (Enseignants, Étudiants, Bâtiments...) seront accessibles. C'est lui qui rend l'application générique.
    \item \textbf{L'Administrateur d'Établissement (ex: SG)} : C'est le gestionnaire local. Il gère les accès des utilisateurs de son école, crée les modèles de documents (templates) et définit la charte graphique. Il supervise la cohérence des données de son institution.
    \item \textbf{L'Utilisateur Métier (ex: CDD, Resp. Master, Agent RH)} : Son rôle est opérationnel. Il utilise les modules qui lui sont assignés pour saisir des données, chercher un bénéficiaire (qu'il s'agisse d'un étudiant ou d'un enseignant), vérifier la prévisualisation et générer les documents officiels.
\end{itemize}

\section{Le dictionnaire de SIRH-Doc}

Pour qu'on se comprenne bien, j'ai dû clarifier quelques concepts clés que j'ai utilisés tout au long du projet :
\begin{itemize}
    \item \textbf{Famille de documents} : C'est une catégorie (ex: \og Scolarité \fg{} ou \og RH \fg{}). Elle définit d'où proviennent les données des bénéficiaires.
    \item \textbf{Template (Modèle)} : C'est la matrice du document avec ses zones (en-tête, corps, pied de page) et ses variables.
    \item \textbf{TableViewConfig} : C'est sans doute le concept le plus important. C'est une configuration qui dit au système : \og Pour la table Personnel, affiche seulement le Nom, le Grade et le Service, et rends le Grade modifiable \fg{}. Cela m'a évité de créer des pages en dur pour chaque table.
\end{itemize}

\section{Besoins fonctionnels : Ce que l'app doit faire}

L'application doit couvrir tout le cycle de vie, de la saisie à l'impression :
\begin{enumerate}
    \item \textbf{Sécurité} : Authentification JWT obligatoire et séparation stricte des données par établissement.
    \item \textbf{Gestion RH} : Création et modification des dossiers individuels, suivi des positions (activité, congé) et archivage des pièces (diplômes).
    \item \textbf{Studio d'Édition} : Un éditeur visuel pour dessiner les modèles avec insertion facile de variables et gestion automatique de la pagination A4.
    \item \textbf{Moteur de Fusion} : Remplacement instantané des variables par les vraies valeurs au moment de la génération.
    \item \textbf{Archivage} : Garder un historique de chaque document émis (qui, quand, pour qui).
\end{enumerate}

\section{Besoins non fonctionnels : La qualité derrière le code}

Au-delà des fonctionnalités, j'avais trois exigences personnelles pour ce projet :
\begin{itemize}
    \item \textbf{Ergonomie} : L'interface doit être sobre et efficace. Un agent RH ne doit pas chercher le bouton \og Générer \fg{} pendant dix minutes.
    \item \textbf{Généricité} : Si l'établissement veut ajouter une table \og Véhicules de service \fg{}, le SuperAdmin doit pouvoir le faire en configuration, sans que je n'aie à modifier le code Angular.
    \item \textbf{Maintenabilité} : Le code doit être découpé proprement (Services, Repositories, DTOs) pour que le projet puisse évoluer après ma soutenance.
\end{itemize}
